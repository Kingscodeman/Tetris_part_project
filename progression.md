# 專案演化歷程：Tetris

這份教學文件將帶領你了解 Tetris（俄羅斯方塊）專案是如何從零開始，逐步發展成一個完整的遊戲。我們會將整個開發過程分為三個階段，說明每個階段新增的程式碼邏輯，並藉由 UML 關係與系統流程，幫助你建構逐步開發的立體概念。目前專案涵蓋了完整的三個階段規劃與實作方向。

---

## 第一階段：基本類別與棋盤骨架

在開發任何圖形化遊戲時，第一步通常是「打地基、架舞台」。針對這點，我們首先要生出遊戲視窗，畫出 10x20 的標準棋盤格線，定義好方塊的資料結構，並在畫面上顯示一個不動的方塊作為測試。

### 📌 新增的功能
* **遊戲視窗建立**：建立可見的 Windows 視窗 (JFrame)。
* **繪製背景與棋盤**：繪製深色背景，畫出 10x20 的格子線。
* **方塊資料結構**：定義出 7 種基本形狀（I, J, L, O, S, T, Z）及其顏色。
* **靜態方塊繪製**：在棋盤上方隨機產生一個不動的方塊，測試繪圖邏輯。

### 🧩 涉及的主要類別
1. **Tetris (視窗入口)**：包含 main 方法，負責建立應用程式視窗。
2. **TetrisPanel (遊戲主面板)**：負責「畫面」邏輯，將格子線與方塊畫出來。
3. **Tetromino (方塊實體)**：代表特定方塊，紀錄現在的 X/Y 座標與旋轉角度。
4. **TetrominoType (方塊類型定義)**：以 Enum 定義 7 種方塊的形狀陣列與顏色配置。

### 🔗 UML 關係 (has-a / is-a)
* **is-a (繼承)**
  * TetrisPanel **is-a** JPanel：面板繼承了 Java Swing 的 JPanel 特性，所以能放上視窗並實作自訂繪圖。
* **has-a (組合/聚合)**
  * Tetris **has-a** TetrisPanel：主視窗擁有一個面板物件。
  * TetrisPanel **has-a** Tetromino (currentPiece)：面板持有一個用來測試的靜態方塊物件。
  * Tetromino **has-a** TetrominoType：每個方塊被實例化時，都綁定特定的型態（如 T 型或 L 型）。

### 🎮 使用案例與情境流程
* **玩家操作情境**：此階段未加入鍵盤操作，玩家僅能啟動遊戲觀看靜態的 10x20 棋盤與上方的一個隨機方塊。
* **系統流程**：
  1. Tetris 的 main 方法啟動視窗。
  2. 初始化 TetrisPanel，隨機生成一個 Tetromino 綁給 currentPiece。
  3. paintComponent 被呼叫，畫出背景框線，並透過形狀陣列畫出目前的 currentPiece。

## 第二階段：移動、碰撞、消橫列與計分

打好地基後，這個階段我們要「讓遊戲動起來」。我們引進定時器讓方塊自動下落，加入鍵盤操作，並實作最核心的遊戲規則：撞到牆壁會停住、連成一直線會消除並且加分。

### 📌 新增的功能
* **自動下落機制**：引入計時器 (`Timer`)，讓方塊隨時間固定向下位移一格。
* **鍵盤控制**：利用 Swing 的 `KeyBindings` 加入左右移動、向上鍵旋轉、向下軟掉落 (Soft Drop) 與空白鍵硬掉落 (Hard Drop) 功能。
* **碰撞偵測 (Collision Detection)**：透過陣列界限運算，確保方塊在此刻或旋轉後的未來，都不會超出 10x20 邊界，也絕不會與已經固定的方塊重疊。
* **方塊鎖定與陣列管理**：方塊觸底且無法再向下移動時，便將其「定型」，把這個方塊的專屬顏色寫入紀錄狀態的 `board` 二維陣列中。
* **消去橫列與計分**：每次方塊鎖定後，程式會由下往上檢查 `board` 是否有哪一整排被非空顏色填滿，若滿排則消除、把上方的堆積方塊「拉下來填補」，並在後台累加消除行數與分數。

### 🧩 涉及的主要類別
1. **TetrisPanel**：此階段開發心力完全集中此處。新增了 `Timer` 物件打造遊戲迴圈，實作按鍵綁定 (`InputMap` 與 `ActionMap`) 來攔截鍵盤，並加入 `Color[][] board` 二維陣列正式啟用鎖定與消除邏輯。

### 🔗 UML 關係 (has-a / is-a)
* **has-a (組合/聚合)**
  * TetrisPanel **has-a** Timer：面板現在擁有一個計時器，充當了遊戲運作的心跳。
  * TetrisPanel **has-a** Color[][] board：面板利用一個二維顏色陣列，作為「死掉固化的積塊們」的永久記憶體。

### 🎮 使用案例與情境流程
* **玩家操作情境**：開啟遊戲後，方塊立刻會自動且穩定地往下落。玩家可以盡情敲擊鍵盤的「方向鍵」旋轉與移動落點。如果確信落點，可按下「空白鍵」直接讓方塊秒掉到底。一旦把整排鋪滿，該橫列會瞬間消失！
* **系統流程**：
  1. **遊戲迴圈起步**：`Timer` 開始計時，每 500 毫秒會呼叫一次 `gameStep()` 方法做為心跳。
  2. **主動位移與鎖定**：`gameStep()` 會主動把 `currentPiece` 往 y 軸 +1 的方向送。碰到下方障礙物時：
     * 將此 `currentPiece` 的每一格座標轉成顏色鎖死寫入 `board`。
     * 對 `board` 陣列做掃描，滿排就做移位消去陣列元素、並加分。
     * 最後從天花板重生一個新的 `Tetromino` 指派給 `currentPiece`。
  3. **玩家輸入介入**：玩家按鍵時不會直接變更畫面，而是會觸發封裝好的任務。系統會先將參數丟入 `isValidPosition()` 方法偵測未來是否會撞牆，只有回傳 `true`，方塊模型的值才會真的被修改，最後呼叫 `repaint()` 通告重新畫圖。

## 第三階段：系統優化與完整遊戲體驗

在方塊可以順利移動、堆疊與消除後，遊戲的核心「引擎」就差不多了。最後這一步是要大幅提升「遊玩體驗 (User Experience)」。我們會加入幫助預測的 UI、遊戲狀態機 (Game State Machine)、以及音效等完整功能，讓專案從「作業」變成「真正的遊戲」。

### 📌 新增的功能
* **下一個方塊預覽 (Next Piece Preview)**：在遊戲迴圈中提早產生下一個方塊，並顯示在畫面的外圍。
* **影子預判方塊 (Ghost Piece)**：讓玩家看見目前的方塊如果掉落到底部會卡在哪裡（通常以半透明狀呈現），大幅降低誤判機率。
* **暫停、重開與 Game Over 狀態**：加入布林值控制器來切換遊戲的進行狀態。若方塊堆積到天花板，則遊戲結束。
* **UI 面板拆分或疊加文字**：在畫面上清楚顯示當前分數 (Score)、消除列數 (Lines) 以及關卡難度速度 (Level) 的字樣。
* **音效整合 (Sound Effects)**：將遊戲過程中的移動、旋轉、消除列、Game Over 對應至獨立的合成器或背景音樂中。

### 🧩 涉及的主要類別
1. **TetrisPanel (或新拆分的 UI 面板/狀態機)**：為避免 `TetrisPanel` 程式碼過於臃腫，會開始切分出「狀態列面板」或是新增狀態控制參數如 `isGameOver`。
2. **SoundTest (或 SoundManager)**：將之前測試過的 Java MIDI 音效類別變成單例 (Singleton) 或可綁定的音效物件，放進遊戲迴圈。

### 🔗 UML 關係 (has-a / is-a)
* **has-a (組合/聚合)**
  * TetrisPanel **has-a** Tetromino (nextPiece)：面板不僅擁有 `currentPiece`，現在也擁有準備生成的預告方塊 `nextPiece`。
  * TetrisPanel **has-a** SoundManager：遊戲畫面面板裝載了一個音效控制員，專門播放各種碰撞音。
  * Tetris (主視窗) **has-a** SidePanel：如果選擇把分數版拆分，則主畫面的右側會加上一個負責顯示文字資訊的新 JPanel 實體。

### 🎮 使用案例與情境流程
* **玩家操作情境**：
  玩家進入遊戲時，一眼就能看到「Game Start」的歡迎詞，遊戲右側清楚顯示分數`0`以及「下一個圖形」。在操作時，畫面下面隨時都有個半透明「影子」提示落點，讓玩家敢大膽地一直按「空白鍵」秒墜！中途如果有事離開，按「P」可以暫停遊戲，文字提示出現。如果不幸滿到頂端，巨大紅色的「Game Over」蓋過畫面，並發出「登～登～登」的音效。按下「Enter」就能再戰一局，分數歸零。
* **系統流程**：
  1. **影子渲染與預覽交替**：
     * `paintComponent` 時，在畫實體方塊之前，先跑一個虛擬迴圈取得該方塊最底部的 Y 軸，再利用 `AlphaComposite` 將該位置畫成半透明，做為 Ghost Piece。
     * 方塊鎖定後，把 `currentPiece` 更新為 `nextPiece` 的值，接著再 `new` 一個全新的 `nextPiece`。
  2. **判斷遊戲結束**：在重生新方塊到頂端座標時，會立刻呼叫 `isValidPosition()` 檢查。如果剛長出來的地方就與固定方塊（`board`）重疊，就把 `isGameOver` 狀態設為 `true`。
  3. **跳過事件迴圈 (State Checking)**：在所有的鍵盤事件任務與 `Timer.gameStep()` 開頭，都會加上 `if (isGameOver || isPaused) return;` 檢查。確保在暫停或結束時，不會進行落下的計算或移動的重繪。
  4. **觸發音效器**：每當 `clearCompletedLines()` 確定橫列消失，或 `gameStep` 確定判定 Game Over 時，會非同步 (Thread) 的方式呼叫 `SoundManager.playTone(...)` 播放相對應的 MIDI 音效。
